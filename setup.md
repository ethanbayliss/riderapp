### WSL wrapper scripts
Flutter looks for tools without `.exe` extension. Wrapper scripts were created at the
SDK paths so Flutter finds them directly.

**`$ANDROID_HOME/platform-tools/adb`**
```bash
#!/bin/bash
exec "$(dirname "$0")/adb.exe" "$@"
```

**`$ANDROID_HOME/emulator/emulator`**
```bash
#!/bin/bash
exec "$(dirname "$0")/emulator.exe" "$@"
```

**`$ANDROID_HOME/build-tools/36.1.0/<tool>`** (aapt, aapt2, zipalign, etc.)
Each tool has a wrapper generated by:
```bash
for exe in "$BUILD_TOOLS_DIR"/*.exe; do
  tool=$(basename "$exe" .exe)
  printf '#!/bin/bash\nexec "$(dirname "$0")/%s.exe" "$@"\n' "$tool" > "$BUILD_TOOLS_DIR/$tool"
  chmod +x "$BUILD_TOOLS_DIR/$tool"
done
```

**`$ANDROID_HOME/cmdline-tools/latest/bin/sdkmanager`** and **`avdmanager`**

These call `.bat` files via `cmd.exe`. Two layers are needed:
1. A shell script wrapper (executed by Linux) that calls cmd.exe from `/mnt/c` to avoid UNC path errors
2. A `.bat` wrapper that sets `JAVA_HOME` to the Windows Android Studio JBR before calling the real tool

Shell script (`cmdline-tools/latest/bin/sdkmanager`):
```bash
#!/bin/bash
cd /mnt/c
exec cmd.exe /c "C:\Users\Ethan\AppData\Local\Android\Sdk\cmdline-tools\latest\bin\sdkmanager.bat" "$@"
```

Batch file (`cmdline-tools/latest/bin/sdkmanager.bat`):
```bat
@echo off
set "JAVA_HOME=C:\Program Files\Android\Android Studio\jbr"
call "C:\Users\Ethan\AppData\Local\Android\Sdk\cmdline-tools\latest-2\bin\sdkmanager.bat" %*
```

> Note: Android Studio installed cmdline-tools to `latest-2` instead of `latest`.
> The `latest/` directory was created manually to hold our wrappers pointing to `latest-2/`.

### Accepting Android SDK licenses
Interactive `flutter doctor --android-licenses` doesn't work well through WSL pipes.
Instead, accepted by piping `y` directly to `sdkmanager --licenses`:
```bash
cd /mnt/c && printf 'y\ny\ny\ny\ny\ny\ny\n' | cmd.exe /c "C:\Users\Ethan\AppData\Local\Android\Sdk\cmdline-tools\latest\bin\sdkmanager.bat --licenses"
```
Some license hashes were also written directly to `$ANDROID_HOME/licenses/`.

### Gradle toolchain workaround (Gradle 8.14 + Ubuntu OpenJDK bug)

Gradle 8.14's toolchain probe incorrectly rejects Ubuntu's OpenJDK 17 at its real path,
even though `javac` works fine. Workaround: create a symlink at a different path and point
Gradle's installation registry there.

```bash
ln -sfn /usr/lib/jvm/java-17-openjdk-amd64 ~/.jdk17
```

In `android/gradle.properties`:
```properties
org.gradle.java.home=/usr/lib/jvm/java-17-openjdk-amd64
org.gradle.java.installations.paths=/home/ethan/.jdk17
org.gradle.java.installations.auto-download=false
```

The symlink makes Gradle treat it as an "external" installation and use the subprocess probe
path instead of the in-process detection, which succeeds.

### Android SDK platform fix

Android Studio may install `android-36.1` but leave `android-36` incomplete (missing `android.jar`
and other files). Flutter targets `android-36`. Fix by copying missing files from `android-36.1`:

```bash
for f in android.jar android-stubs-src.jar build.prop core-for-system-modules.jar; do
  cp "$ANDROID_HOME/platforms/android-36.1/$f" "$ANDROID_HOME/platforms/android-36/$f"
done
```

### Known issues / gotchas
- `cmd.exe` does not support UNC paths — always `cd /mnt/c` before calling `cmd.exe` from WSL
- `sdkmanager.bat` needs a Windows `JAVA_HOME` (not the WSL one) — use Android Studio's bundled JBR
- First `flutter run` build is slow (10-15 min) due to Gradle downloading dependencies over the `/mnt/c` filesystem
- If emulator shows as `offline` in `adb devices`, Android Studio has a conflicting ADB server — run `adb kill-server && adb start-server`
